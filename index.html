Section 01 · Introduction

Why Does This Change Everything?

Picture a vending machine. You insert a coin, press a button, and out comes your snack. The machine runs on pure, rigid rules — it can't smell if the snack is stale, it can't read a news article, and it can't make a judgment call.

Traditional blockchain smart contracts are exactly like that vending machine. They are deterministic, meaning they can only compute things that have a single, predictable answer. Ask "is 2 + 2 = 4?" — perfect. Ask "Is this news article credible?" — impossible. They have no brain.

GenLayer changes this entirely. It is an AI-native blockchain where smart contracts — called Intelligent Contracts — are written in Python and can call Large Language Models (LLMs), fetch live data from the web, and make judgment calls. All of this happens on-chain, trustlessly, without a centralized oracle service.

🚀 What We're Building
An "AI Yes/No Oracle" — a dApp where anyone can ask a question, and a Python Intelligent Contract uses AI to answer YES or NO. The answer is stored on-chain and readable by anyone, forever.

By the end of this tutorial, you will understand how GenLayer works, have written a real Intelligent Contract, deployed it in the browser-based Studio, and built a working JavaScript frontend. Let's go.

Section 02 · Core Concepts

Two Big Ideas That Power Everything

Before writing a single line of code, you need to understand two concepts. Don't worry — they're actually quite intuitive once you see the right analogy.

Optimistic Democracy

Imagine a court case. When a defendant walks into court, the legal system presumes they are innocent — optimistically. A panel of diverse judges independently review the evidence. They each reach a verdict. If the majority agree, that verdict becomes law.

GenLayer works exactly the same way. When your contract calls an LLM, a randomly selected group of validators each run the contract independently, each using their own AI model (GPT, LLaMA, Mistral, etc.). One validator is chosen as the Leader — they propose the result first. The other validators then check whether the Leader's result is acceptable.

⚖️

The Leader

One validator runs first and proposes the result. Like a judge writing the first draft of a verdict.

🗳️

The Validators

Other validators independently verify the result. They vote: is the Leader's answer good enough?

💡 The Key Insight
The "optimistic" part means the system assumes the Leader is honest and accepts the result quickly. Validators only challenge if something looks wrong. This makes GenLayer fast, while remaining trustless.

The Equivalence Principle

Here's a problem: if five validators each ask an LLM "Is the sky blue?", they might get five slightly different worded answers — "Yes", "Definitely yes", "Affirmative", "Of course!", "The sky is indeed blue." They all mean the same thing, but they're not identical strings.

The Equivalence Principle is your answer to this. It's a rule you define in your contract that tells validators what counts as agreement. GenLayer gives you three tools:

Mode When to Use It Example
strict_eq Results must be byte-for-byte identical Boolean True/False, exact numbers
non_comparative Each validator just checks its own result against a rule "Does this text contain the word 'yes'?"
comparative Use a second LLM call to check if two results are equivalent Long-form text where wording varies

For our Yes/No Oracle, we'll use strict_eq — our contract will return a clean boolean (True for YES, False for NO), so validators can agree on an exact match easily.

Line-by-Line Walkthrough

# { "Depends": "py-genlayer:test" }

This comment at the top is actually a configuration header. It tells GenVM which SDK version to load. Think of it like a requirements.txt for your contract. Keep it exactly as shown.

from genlayer import *

Imports the entire GenLayer SDK. This gives you access to gl.Contract, gl.exec_prompt, gl.eq_principle_strict_eq, and all state variable types.

State Variables: questions, answers, question_count

These are declared at the class level with type annotations. Unlike a regular Python class, these are persisted to the blockchain's state trie — they survive between calls, forever. gl.DynArray is a dynamic array type that GenLayer knows how to serialize and store on-chain.

def call_ai(): — the Non-Deterministic Block

This inner function is the heart of the contract. It's a plain Python function with no arguments. Inside, we call gl.exec_prompt(prompt) which routes to the validator's configured LLM. The key insight: this function runs independently on each validator node. Each validator sends the prompt to their own AI model and gets a response.

gl.eq_principle_strict_eq(call_ai)

This is GenLayer's magic wrapper. It takes the inner function, runs it inside the non-deterministic sandbox, and applies the Equivalence Principle. The strict_eq variant means: all validators must return the exact same boolean. Our prompt enforces this by asking the LLM to output only "YES" or "NO" — making the result a clean True/False that validators can match exactly.

🧠 Why strict_eq works here
Because we parse the LLM's response into a Python boolean inside the non-deterministic block, the result passed to strict_eq is always True or False. There's no ambiguity. Validators with different LLMs answering "YES" and "Yes, definitely!" both resolve to True, and they agree.

View Methods

Methods decorated with @gl.public.view are read-only — they don't change state and don't require a transaction (or gas). @gl.public.write methods change state and go through the consensus process.

const client = createClient({
chain:   simulator,

// ── WRITE: Ask the Oracle a Question ───────────────────────────────────
async function askQuestion(question) {
console.log(`📨 Sending question to the oracle: "${question}"`);

// Initialize consensus smart contract (required before first interaction)
await client.initializeConsensusSmartContract();

// Send the write transaction to the contract
const txHash = await client.writeContract({
address:      CONTRACT_ADDRESS,
functionName: ‘ask_question’,
args:         [question],
value:        0n,           // No GEN tokens needed
});

console.log(`⏳ Transaction submitted: ${txHash}`);
console.log(‘🔄 Waiting for AI validators to reach consensus…’);

// Wait for validators to finalize the transaction
const receipt = await client.waitForTransactionReceipt({
hash:     txHash,
status:   TransactionStatus.FINALIZED, // Wait for full consensus
retries:  100,                        // Try up to 100 times
interval: 3000,                       // Every 3 seconds
});

console.log(‘✅ Transaction finalized!’, receipt);
return receipt;

What Happens at Each Step

  1. 1
    User types a question in the HTML input and clicks "Ask". JavaScript calls askQuestion().
  2. 2
    genlayer-js signs and sends a write transaction to the GenLayer network, targeting your contract's ask_question function.
  3. 3
    GenVM runs the contract on the Leader validator. The call_ai() block executes — the LLM prompt is sent to the validator's AI model. It returns "YES" or "NO".
  4. 4
    Other validators independently verify. Each runs the same non-deterministic block with their own AI. They check: does the Leader's boolean (True/False) match what they got? If majority agree → consensus reached.
  5. 5
    The answer is committed to state. The questions and answers arrays are updated on-chain, permanently. Transaction status becomes FINALIZED.
  6. 6
    Frontend reads the result. getLatestAnswer() calls get_latest() — a free view call that reads the chain state. Displays YES ✅ or NO ❌ to the user.

Section 09 · Why GenLayer

What Makes This Genuinely Powerful

1. AI Lives Inside the Contract

On every other blockchain, if you want AI, you must run it off-chain, trust a centralized API, and pipe the result back in via an oracle. With GenLayer, the AI call is the contract logic. There's no off-chain server, no API key, no single point of failure. The LLM call happens inside GenVM, and the result is verified by the consensus layer.

2. No Oracle Needed

Chainlink, Band Protocol, API3 — oracle networks exist because blockchains can't access the real world. GenLayer makes oracles obsolete for AI-answerable questions. Your contract IS the oracle. And unlike traditional oracles, it's not controlled by one entity — it's verified by a decentralized network of AI-running validators.

3. Real-World Use Cases

ApplicationHow GenLayer Enables It
Prediction Markets Contract reads live web data, AI determines the outcome of real-world events
AI Content Moderation On-chain AI evaluates submitted content against community guidelines
Dispute Resolution Validators with diverse LLMs reach consensus on subjective disagreements
Insurance Claims Contract verifies claim validity by fetching and reasoning over real-world data
DAO Governance AI evaluates whether proposals align with community values — on-chain
AI Agents Autonomous agents that can transact and make decisions trustlessly
🌐 The Bigger Picture
Bitcoin gave us trustless money. Ethereum gave us trustless computation. GenLayer gives us trustless decision-making. This is the missing piece for the AI age — a way to make AI outputs enforceable, verifiable, and permanent on-chain.
